# Pages Gallery Console 设计

## 目标

对当前 GitHub Pages 预览页做一次完全重设计。

这次不再基于“后台检视台”思路继续修补，而是改成一个更接近高端数字画廊的内容预览页：

- 图片是绝对主角
- JSON 是辅助策展信息
- 日期切换是时间轴导航

同时仍然保留这个页面的实用目标：

1. 快速切换不同日期
2. 优先完整查看正式图片
3. 查看关键信息和原始 JSON

## 设计方向

### 风格名称

Gallery Console

### 风格关键词

- 高级数字画廊
- 策展型内容预览
- 大图主舞台
- 克制但有戏剧性
- 统一画布而不是三块后台卡片

### 为什么选这个方向

当前页面虽然已经具备基本检视能力，但它仍然更像：

- 一个功能页面
- 一个工具台

而不是一个真正“为视觉产物服务”的预览空间。

这个项目里的正式图片本身已经有较强视觉风格，所以预览页不应该继续像后台，而应该像：

- 一个展示正式产物的策展页
- 一个带数据辅助的高质量查看器

因此这次的页面重设计要从“工具面板思维”切换到“展厅思维”。

## 核心原则

### 1. 图片优先

无论任何设备、任何分辨率：

- 图片始终是页面第一视觉焦点
- 图片必须优先完整显示
- 其他区域不能反过来挤压图片主舞台

### 2. 信息次级

JSON 不再作为和图片平权的大区块。

它应该拆成两层：

1. 关键信息摘要
2. 原始 JSON 抽屉 / 可展开区

### 3. 统一画布

不再维持“三个大盒子并排”的分裂结构。

页面需要像一整张统一画布，然后在其中放置：

- 图片主舞台
- 时间轴
- 信息层

区分通过：

- 层级
- 透明度
- 细边框
- 阴影
- 位置关系

而不是三种完全不同的背景块。

## 页面结构

### 桌面端

桌面端采用“主舞台 + 辅助层”的结构，而不是简单三列。

建议结构：

1. 左侧或底部细时间轴
2. 中央主图片舞台
3. 右侧信息层

其中：

- 图片舞台占页面最大区域
- 右侧信息层宽度受控
- 时间轴更轻，不抢主内容

### 平板端

平板端保持图片主舞台为核心：

- 图片仍为首屏主体
- 信息层压缩为更窄的侧栏或下方块
- 时间轴改成更紧凑的列表或横向带

### 手机端

手机端优先规则不变：

- 首屏优先完整看图片
- 下方展示关键信息
- 原始 JSON 放在可展开区域

因此手机端结构建议为：

1. 顶部轻量工具带
2. 中部图片主舞台
3. 下部信息区
4. JSON 抽屉 / 展开块

## 视觉层次

### 背景

背景不是简单的纯色，而是统一氛围层：

- 深墨黑 / 深绿黑渐变
- 轻微灯光感
- 一点颗粒或柔和纹理

目的是让图片像被陈列在一块高级背景上。

### 图片舞台

图片舞台应该像画廊展框：

- 大面积留白或留黑
- 图片居中
- 轻浮层阴影
- 不要多余边框干扰图片

页面应该看起来像是：

- 图片被“挂”在页面中央
- 而不是“塞”进一张卡片里

### 信息区

信息区应该更像策展说明牌，而不是代码面板。

推荐分成两层：

1. 顶部：摘要信息
   - 日期
   - 标题
   - 发布时间
   - quote
   - 来源

2. 底部：原始 JSON
   - 折叠或独立滚动区域
   - 默认不抢眼

### 时间轴

日期导航应从“按钮列表”升级为“时间轴”：

- 当前日期高亮更像展签
- 其他日期为次级刻度
- 桌面端可竖向
- 手机端可横向

## 字体策略

### 标题

标题采用更有策展气质的 serif：

- `Iowan Old Style`
- `Songti SC`
- `Noto Serif SC`

### UI 文本

界面文本用 clean sans：

- `Avenir Next`
- `PingFang SC`
- `Hiragino Sans GB`

### 数据文本

原始 JSON 保持 monospace，但面积更克制，避免成为主角。

## 色彩策略

推荐方向：

- 背景：深墨黑绿
- 文本：骨白 / 暖白
- 强调：铜金或冰蓝

本版优先使用铜金：

- 更符合“展厅 / 策展”气质
- 比科技蓝更有人味

建议配色：

- 背景：`#101514`
- 深层面：`#171d1d`
- 主文本：`#f4ede3`
- 次文本：`rgba(244, 237, 227, 0.66)`
- 强调：`#b57a3f`
- 线条：`rgba(255, 246, 232, 0.12)`

## 交互设计

### 原图 / 原 JSON

仍然保留：

- 打开原图
- 打开原 JSON

但它们不应该是大而重的按钮，而是更轻的动作链接或小型控制。

### JSON 展开

原始 JSON 区不建议默认完全展开占大面积空间。

建议：

- 默认显示摘要信息
- 原始 JSON 放在可展开详情区

这样图片的视觉主导地位更稳。

### 状态反馈

错误或空状态需要更有设计感：

- 图片缺失：舞台区显示空状态文案
- JSON 缺失：信息区给出提示
- 索引为空：页面显示统一的空状态，而不是零散报错文本

## 响应式原则

### 桌面端

- 图片始终最大
- 信息层次围绕图片组织

### 平板端

- 时间轴压缩
- 信息区收窄
- 图片仍优先完整显示

### 手机端

- 图片优先
- 信息区在下方
- JSON 可展开或独立滚动

### 所有设备共同要求

- 不允许页面让图片因信息区过大而失去主体地位
- 不允许原始 JSON 成为视觉最大块

## 与当前页面的主要差异

当前页面的问题：

1. 仍然是工具页面思路
2. 三块结构太明显
3. JSON 占位感过强
4. 日期切换还是“按钮区”，不够像时间轴

这次重设计后的变化：

1. 从工具台切到策展展示台
2. 图片成为真正的中心
3. 信息从大代码面板降级为辅助层
4. 页面整体从“三块拼接”变成“一张统一画布”

## 成功标准

完成后，页面需要满足：

1. 第一眼先看到图片，而不是 JSON 或控制区
2. 页面整体看起来像一个高端内容预览器，而不是后台
3. 不同设备下都保持图片完整优先
4. JSON 仍然可查，但不再抢主舞台
5. 日期切换更像时间轴，而不是按钮列表

## 实施范围

本次只改：

- `pages/index.html`
- `pages/styles.css`
- `pages/app.js`
- 对应前端契约测试

本次不改：

- 后端渲染逻辑
- JSON 数据结构
- 图片生成逻辑
- GitHub Pages 部署方式
